# 物流批量下单导入系统项目说明书

## 一、项目背景

物流/快递行业在日常业务中存在大量批量下单需求，客户通常通过 Excel 方式提交寄件信息。但由于不同客户使用的模板格式不统一，常出现列名不同、列顺序不同、字段缺失、合并单元格等情况，导致人工整理和逐条录入效率低、出错率高，严重影响下单时效。

为提升批量下单效率，降低人工处理成本，现计划开发一套 Web 应用，支持多种 Excel 模板的自动识别与导入，并完成批量下单流程。系统部署至 Vercel 平台，支持在线访问与使用，满足业务快速接入和灵活扩展的需求。

## 二、技术要求

系统需支持对不同 Excel 模板的自动识别与导入，能够兼容列名不同、列序不同、存在合并单元格等情况，并将数据映射到统一订单字段。导入时需进行字段校验、格式校验、必填校验与去重处理，支持批量下单。

| 字段名 | 说明 | 必填 |
|---|---|---|
| 外部编码 | 外部系统订单唯一编号，用于去重 | 否 |
| 发件人姓名 | 寄件人姓名 | 是 |
| 发件人电话 | 寄件人联系方式 | 是 |
| 发件人地址 | 寄件人完整地址 | 是 |
| 收件人姓名 | 收货人姓名 | 是 |
| 收件人电话 | 收货人联系方式 | 是 |
| 收件人地址 | 收货人完整地址 | 是 |
| 重量(kg) | 货物重量，必须为正数 | 是 |
| 件数 | 包裹数量，必须为正整数 | 是 |
| 温层 | 常温 / 冷藏 / 冷冻（可选其一） | 是 |
| 备注 | 附加说明 | 否 |

## 三、系统架构

系统采用前后端一体化 Web 架构，基于 `Next.js App Router + TypeScript` 实现，部署于 Vercel。

### 1. 前端

- 基于 Next.js App Router 构建页面与交互
- 负责 Excel 上传、模板识别结果展示、手动映射、数据预览编辑、错误提示、导出、提交操作
- 使用 XLSX 库进行前端 Excel 解析与导出

### 2. 后端

- 使用 Next.js Route Handlers 提供接口
- 提供历史运单查询接口
- 提供批量提交下单接口
- 负责与数据库进行读写交互

### 3. 数据存储

- 通过 Vercel Marketplace 集成数据库
- 推荐使用 Neon / Supabase / Turso / 兼容 PostgreSQL 的数据库
- 运单数据统一写入数据库，不允许仅保存在本地内存或本地文件中

## 四、功能需求

### 模块一：模板管理与文件导入

- 支持上传 Excel 文件（`.xlsx` / `.xls`），支持拖拽上传和点击上传
- 支持至少 2 种不同模板格式的自动识别，能够适配不同列名、不同列序
- 支持用户手动选择 Excel 列与系统字段的映射关系
- 支持模板记忆学习：当用户手动调整列映射后，系统自动记录映射规则；下次上传相同或相似结构模板时自动应用
- 导入过程中显示实时进度条，展示百分比及当前处理条数/总条数
- 支持 1000 条以上数据导入，确保不卡顿、不崩溃
- 对文件格式错误、文件为空、编码异常、Sheet 不存在等情况给出明确提示

### 模块二：数据预览与编辑

- 导入后进入预览列表页面，以类 Excel 表格形式展示全部数据
- 表格支持表头固定、横向滚动、列宽合理
- 支持单元格点击直接编辑
- 支持编辑后即时校验
- 必填缺失、电话格式错误、重量/件数非正数、温层值不合法时高亮提示
- 必须一次性展示全部错误，标注行号、字段名和错误原因
- 支持外部编码重复检测，包括同批次内重复和与历史数据重复
- 错误字段支持文字提示
- 支持新增空行、删除行
- 支持将当前预览数据导出为 Excel

### 模块三：提交下单

- 有错误的数据行不允许提交
- 点击“提交下单”后显示提交进度条
- 提交成功后数据持久化到数据库
- 返回提交结果汇总：成功 N 条、失败 N 条

### 模块四：已导入运单列表

- 支持查看所有历史已导入运单记录
- 列表数据从数据库读取
- 支持按外部编码、收件人姓名、提交时间进行筛选与搜索
- 支持分页展示

## 五、非功能需求

- 界面布局清晰，操作路径直观
- 支持桌面端和移动端访问
- 导入和提交过程具备进度反馈
- 操作具备 Loading、成功、失败提示
- 按钮具备防重复点击能力
- TypeScript 类型完整
- 组件拆分合理，代码结构清晰
- 构建无报错，可正常部署到 Vercel
